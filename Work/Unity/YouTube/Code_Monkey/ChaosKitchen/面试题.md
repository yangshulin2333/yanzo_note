# “你如何处理不同游戏模块之间的交互？”


> “在处理复杂的游戏对象交互时，我倾向于使用**观察者模式**。 比如在《混乱厨房》项目中，我通过 C# 的 `EventHandler` 实现了逻辑层（如炉灶计时）与表现层（如音效、视觉特效）的彻底解耦。 广播者只负责在状态改变时通过 `?.Invoke` 发出通知，而不需要知道谁在监听。 为了保证系统的健壮性和内存安全，我采用了严谨的生命周期管理：在 `Start` 中使用 `+=` 登记，在 `OnDestroy` 中使用 `-=` 注销。 这种做法不仅避免了 Unity 常见的空引用异常（NullReferenceException），更防止了因为委托强引用导致的**内存泄漏**。这跟我之前处理医疗系统数据时的闭环思维是一致的，即：**有始有终，所有申请的资源引用都必须有明确的释放点。**”

### 最后的背诵清单（出门面试前看一眼）：

1. **定义**：`event` 关键字，底层是 `MulticastDelegate`（多播委托）。
    
2. **订阅**：`+=`，把方法指针和对象引用（Target）塞进 `Invocation List`。
    
3. **触发**：`?.Invoke`，遍历列表，逐个执行方法。
    
4. **注销**：`-=`，从列表中移除引用，切断“强引用”联系。
    
5. **后果**：不注销会导致已销毁对象仍被引用，GC 无法回收，产生内存泄漏。
