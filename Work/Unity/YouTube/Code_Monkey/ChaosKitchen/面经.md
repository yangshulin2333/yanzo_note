# 第一模块：《Kitchen Chaos》项目深度剖析（简历核心）

面试官看这个项目，不是看你能不能跟着教程做完，而是看你**是否理解了现代游戏开发的架构体系**。你需要把这个项目包装成一个“高度解耦、数据驱动”的优秀工程。

### 1. 核心架构：事件驱动（Event-Driven）与观察者模式（Observer Pattern）

- **知识点本质**：项目中大量使用了 `C# event EventHandler` 和 `Action`。
    
- **背诵话术**：“在这个项目中，我为了避免各个系统之间的高耦合，大量采用了观察者模式。比如，当玩家选中一个柜台时，如果直接在玩家脚本里去调用柜台的高亮特效代码，会导致 Player 和所有 Counter 产生强依赖。我的做法是：在 Player 身上定义一个 `OnSelectedCounterChanged` 事件。玩家只负责‘广播’当前选中的目标是谁。而各个柜台的 `Visual`（视觉）脚本在 `Start` 生命周期中去‘订阅’这个事件。这样，逻辑层（数据）和表现层（视觉）实现了完全分离。未来如果我要加一个音效系统，只需要让音效脚本也去订阅这个事件即可，完全不需要修改 Player 的代码。这符合面向对象的**开闭原则（对扩展开放，对修改封闭）**。”
    

### 2. 数据管理：Scriptable Object (SO) 数据驱动设计

- **知识点本质**：使用 SO 替代 JSON/XML 管理静态数据。
    
- **背诵话术**：“对于游戏内的静态数据，比如番茄、肉饼等 `KitchenObject` 的预制体、图标、名称，以及切菜台的 `CuttingRecipe`（比如一个番茄切几刀变成番茄片），我全部使用了 Scriptable Object 来进行配置。
    
    - **优势 1：内存优化。** 如果把这些属性写在 MonoBehaviour 里，每一个实例化的番茄都会在内存中保存一份相同的数据。而使用 SO，无论场景里有多少个番茄，它们引用的都是内存中同一份 SO 数据，极大地降低了内存占用。
        
    - **优势 2：工作流解耦。** 策划或美术可以直接在 Project 窗口修改 SO 的配置（比如调整切菜所需的刀数），而不需要去动代码或者寻找场景里的预制体，实现了数据和逻辑的分离。”
        

### 3. 多态与接口：`IKitchenObjectParent` 的精妙运用

- **知识点本质**：接口（Interface）在实际业务中的运用。
    
- **背诵话术**：“在厨房逻辑中，有一个核心痛点：‘食材’（KitchenObject）可以放在‘玩家’手上，也可以放在‘各种柜台’上。为了统一管理食材的转移，我抽象出了一个 `IKitchenObjectParent` 接口。
    
    - 这个接口定义了 `GetKitchenObject()`、`SetKitchenObject()`、`ClearKitchenObject()` 等规范。
        
    - 让 Player 和 BaseCounter 都去实现这个接口。这样一来，当食材在不同载体之间传递时，食材自身的代码根本不需要知道自己是在人手上还是在桌子上，它只需要调用父级的接口方法即可。这利用了**多态**的特性，极大简化了物品交互的逻辑，如果以后增加传送带或服务员，只要实现这个接口就能直接接入系统。”
        

### 4. 状态机模式（Finite State Machine - FSM）：炉灶逻辑

- **知识点本质**：复杂状态的清晰流转。
    
- **背诵话术**：“对于炉灶柜台（StoveCounter），肉饼的煎烤过程非常复杂。我没有使用多个布尔值（比如 `isFrying`, `isFried`）来控制，而是引入了状态机模式。
    
    - 我定义了一个枚举 `State { Idle, Frying, Fried, Burned }`。
        
    - 在 `Update` 函数中，使用 `switch-case` 根据当前状态执行不同的逻辑（比如在 `Frying` 状态下增加计时器，时间到了自动切换到 `Fried` 状态）。
        
    - **优势**：状态机让代码的意图非常清晰，避免了‘面条式’的 `if-else` 嵌套，并且有效防止了状态冲突（比如不可能同时处于生和焦的状态）。”

# 第二模块：C# 核心底层“八股文”（重中之重）

魂月科技一定会深挖 C# 的底层，因为这是优化游戏的根本。

### 1. 值类型（Value Type）与引用类型（Reference Type）的区别？（必考）

- **内存分配**：
    
    - **值类型**（`int`, `float`, `bool`, `struct`, `enum`）：通常分配在**线程栈（Stack）**上。栈内存由系统自动管理，作用域结束即被释放，效率极高，**不会产生 GC（垃圾回收）**。
        
    - **引用类型**（`class`, `string`, `Array`, `Delegate`, `object`）：它的“数据实体”分配在**托管堆（Heap）**上，而它的“引用地址（指针）”分配在栈上。堆内存由 GC 负责回收。
        
- **传递方式**：值类型按值传递（拷贝一份副本），修改副本不影响原值；引用类型按引用传递（传递内存地址），修改会影响原对象。
    
- **装箱与拆箱（Boxing/Unboxing）**：
    
    - **装箱**：把值类型转换为引用类型（比如把 `int` 赋值给 `object`）。这会在堆上重新分配内存，并把栈上的值拷贝过去，**极度消耗性能并产生内存碎片**。
        
    - **拆箱**：把引用类型转回值类型。
        
    - **面试加分句**：“在实际开发中，应尽量避免装箱拆箱。例如，早期 Unity 的非泛型集合 `ArrayList` 存入 `int` 会发生装箱，现在我们都使用泛型集合 `List<int>` 来避免这个问题。”
        

### 2. `List` 和 `Dictionary` 的底层原理是什么？（高频）

- **`List<T>` 原理**：底层是一个**动态数组**。
    
    - 当添加元素导致容量不足时，它会自动**扩容**，通常是申请一个大小为原来 **2 倍**的新数组，然后将旧数组的元素 Copy 过去。
        
    - **优化考点**：扩容非常耗时且产生大量 GC。因此在实例化 List 时，如果能预估数据量，**一定要指定初始容量** `new List<int>(100)`。
        
- **`Dictionary<TKey, TValue>` 原理**：底层是**哈希表（Hash Table）**。
    
    - 通过对 Key 进行 Hash 算法得到一个 HashCode，再经过取模运算得到数组索引，直接存取数据。
        
    - **优势**：查找的时间复杂度是 **O(1)**，速度极快。
        
    - **哈希冲突**：如果两个不同的 Key 算出了相同的索引，C# 使用“拉链法”（链表）来解决冲突。
        

### 3. GC（垃圾回收机制）的原理与优化？（决定薪资的一道题）

- **GC 触发条件**：当托管堆（Heap）内存不足时，系统会暂停所有线程执行 GC 操作，这会导致游戏卡顿（GC Spike）。
    
- **Generational GC（分代回收）**：C# 的堆分为第 0 代、第 1 代、第 2 代。新创建的对象在第 0 代。每次 GC 存活下来的对象会被升级到下一代。代数越老，GC 检查的频率越低。
    
- **面试话术（如何在 Unity 中避免 GC）**：
    
    1. **使用对象池（Object Pool）**：对于频繁创建和销毁的物体（如子弹、特效、KitchenObject），不要使用 `Instantiate` 和 `Destroy`，而是在游戏开始时创建一批隐藏起来，需要时取出，用完后隐藏。
        
    2. **避免在 `Update` 中使用 `new`**：绝对不要在每帧调用的函数里实例化类或数组。
        
    3. **字符串优化**：String 是不可变的。频繁拼接字符串（如 `str = a + b + c`）会产生大量临时垃圾。应该使用 `StringBuilder`。
        
    4. **避免闭包与 LINQ**：在热更代码或高频调用的地方，慎用 LINQ 和匿名函数，它们在底层容易产生额外的分配。
        

### 4. `ref` 和 `out` 的区别？

- 两者都用于按引用传递参数。
    
- `ref`：传入前**必须被初始化**，函数内可以不赋值。
    
- `out`：传入前**可以不初始化**，但在函数内部**必须被赋值**（常用于函数返回多个值）。

# 第三模块：Unity 引擎进阶“八股文”（体现专业度）

### 1. MonoBehaviour 生命周期与执行顺序（肌肉记忆级别）

- `Awake`：脚本实例被创建时调用，无论脚本是否 enabled 都会执行（只要 GameObject 是 Active 的）。常用于获取组件、初始化。
    
- `OnEnable`：每次脚本被激活时调用。常用于**注册事件**。
    
- `Start`：在所有 `Awake` 执行完毕后的第一帧，`Update` 之前调用。仅执行一次。
    
- `FixedUpdate`：物理帧更新，频率固定（默认 0.02s）。与物理相关的计算（加力、修改 Rigidbody）必须放在这里，不受帧率掉帧影响。
    
- `Update`：逻辑帧更新，每帧执行一次。处理输入、非物理移动。
    
- `LateUpdate`：在所有 `Update` 执行完后调用。**必考点**：绝对常用于摄像机跟随（Cinemachine 底层原理），因为要确保被跟随的目标在 Update 中移动完毕后，摄像机再移动，否则会出现画面抖动。
    
- `OnDisable`：脚本被禁用时调用。常用于**注销事件**（如果不注销会导致内存泄漏）。
    
- `OnDestroy`：对象被销毁前调用。
    

### 2. 协程（Coroutine）的底层原理是什么？它是多线程吗？

- **结论**：**协程绝对不是多线程！** 协程依然运行在 Unity 的主线程上。
    
- **原理**：协程的本质是 C# 的**迭代器（IEnumerator）**和**状态机**。
    
- **解释**：当协程遇到 `yield return` 时，会将当前的执行状态保存下来，将控制权交还给 Unity 的主循环。等到下一帧（或指定时间 `WaitForSeconds` 后），Unity 会根据内部的调度机制，找到这个状态机，从上次 `yield` 的地方继续向下执行。它利用了分时操作，避免了阻塞主线程。
    

### 3. Draw Call 与 渲染优化（图形学基础）

- **概念**：Draw Call 是 CPU 准备好材质、网格等渲染数据后，向 GPU 发出的一条绘制指令。
    
- **为什么 Draw Call 高会卡？**：因为 CPU 准备数据和切换渲染状态（SetPass Call）非常耗时。GPU 画图很快，但 CPU 发号施令慢，导致 GPU 经常在闲置等待。
    
- **优化手段（必背）**：
    
    1. **静态合批（Static Batching）**：勾选场景中不动的物体的 `Static` 属性。Unity 会在构建时将使用**相同材质**的网格合并成一个大网格，一次性交给 GPU。代价是增加内存（包体变大）。
        
    2. **动态合批（Dynamic Batching）**：Unity 自动处理，条件极其苛刻（顶点数少于 300 等）。
        
    3. **GPU Instancing（GPU 实例化）**：针对大量**相同网格和相同材质**的物体（如森林里的树、地上的草、甚至厨房里的大量相同盘子）。CPU 只发送一次网格和材质数据，外加一个包含所有实例位置/旋转的数组，GPU 一次性画出所有物体。
        
    4. **UI 动静分离**：Canvas 重绘（Rebuild）极其耗性能。要把经常变动的 UI（如血条、计时器）和静态 UI（如背景图、边框）分在不同的 Canvas 里，避免一个元素变动导致整个界面的网格重新计算。
        

### 4. 碰撞系统：Collider vs Trigger / Raycast

- **Collider 碰撞**：发生物理反弹。必要条件：双方都有 Collider，且至少一方带有 `Rigidbody`（刚体），且不能勾选 `Is Trigger`。在 `OnCollisionEnter` 响应。
    
- **Trigger 触发**：穿透感应。一方勾选 `Is Trigger`，至少一方有 Rigidbody。在 `OnTriggerEnter` 响应。（比如：厨房柜台前方的感应区，玩家走进去触发交互）。
    
- **射线检测（Raycast）**：从一点向特定方向发射一条不可见的线，返回碰到的第一个（或所有）带有 Collider 的物体。
    
    - **优化策略**：一定要使用 `LayerMask`（层级蒙版）！进行射线检测时，通过 LayerMask 忽略掉不需要检测的层（比如忽略 UI 层、特效层），能成倍减少物理引擎的计算量。在《混乱厨房》中，玩家面向柜台的交互检测就是用 Raycast 配合特定 Layer 实现的。
